# 产品形态推演

## 产品名：判镜（JudgMirror）

不做 SaaS 控制台。用对话界面作为入口，但**前端不止聊天**。

用户感知：一个懂我的内容策略顾问。
产品本质：用 LLM 做交互层，用 rubrics + 决策痕迹 + 模式识别做判断层。

---

## 产品和 LLM 的关系

判镜不是独立 AI。它寄生在 LLM 上。衔接点在三个位置：

### 衔接点 A：输入翻译层（每次交互触发）

用户说话 → LLM 识别"这是一个判断请求" → 调出对应维度和历史权重 → 返回结构化判断。

LLM 做的是**语义→结构**的翻译。用户说"这个能不能做"，LLM 知道这意味着要拆哪些维度，但维度定义和权重来自用户的校准库。

### 衔接点 B：权重调取层（用户历史参与）

用户说"这个选题能不能做"时，LLM 从校准库里调出：
- 这个人过去 50 次选题的预测准确率
- 预测力最高的维度（比如"个人匹配"比"情绪钩子"更准）
- 当前 rubrics 版本权重

LLM 不做推理，只做**检索+组装**。

### 衔接点 C：反思触发层（定期/用户主动）

用户说"复盘"，LLM 调出历史决策痕迹 → 预测 vs 实际对照 → 哪些维度偏了 → 建议调整权重。

LLM 做**模式识别**——在用户的决策历史里找规律。

---

## 输入输出定义

**输入（用户侧）：**
- 一段自然语言——"这个能不能做""我该接这个项目吗"
- 不需要填表单、不需要学概念

**输出（用户看到的）：**
- 维度打分表（可读、可质疑、每项有一句理由）
- 综合判断（做/不做/改方向）
- 可选的下一步动作——"要我爆破吗""要我记录吗"

## 产品前台形态（不止聊天）

### 1. 判断仪表盘

```
本周判断概覽
决策总数: 12
事后已验证: 3
准确率: 2/3 ✓

最准的维度: 标题情绪钩子（预测力 0.8）
最偏的维度: 时效性判断（连续 3 次高估，建议降权）

当前 rubrics 版本: v4.3
上次校准: 3 天前
建议下次校准: 明天（已有 5 条新数据可回收）
```

用户打开的第一眼不是空白对话框，是一面镜子。

### 2. 决策卡片

每次重要判断生成一张独立卡片——可分享、可搜索、可按时间线翻看。决策变成了可触摸的物件，不是聊天框里一段会滚走的文本。

### 3. 推送/提醒

数据回来了主动说。竞品有动作主动说。判断偏了主动说。不是等用户来问。

### 4. 语音/纸质（远离屏幕的决策模式）

重大决策时刻，产品刻意不让你对着屏幕。语音口述→系统结构化→生成卡片。或者打印纸质决策模板→用笔填→拍照→OCR 存档。

屏幕不是答案。思考过程比答案重要。

### 5. 团队校准面板

多人对同一件事的判断分歧 → 结构化分歧维度 → 等数据回来 → 告诉谁这次对了。

## 垂直场景策略：内容创作

### 不打"判断力提升"旗号

用户不觉得"我需要提升判断力"。
用户觉得"为什么我认真做的内容不火，随手发的爆了？"

产品语言：**"帮你搞清楚你到底输在哪"**。

### 切入时机：复盘时刻

不做事前预测（"这条能不能爆"）——那是远期。
做**事后诊断**——用户发了 20 条，贴数据进来，判镜做三件事：

1. **拆因子** — 按维度自动分类和打分
2. **找模式** — "你的高播放 80% 在周二""标题带数字比问句平均播放高 40%""完播率塌点统一在第 15 秒"
3. **给动作** — 下一条该改什么

### 不做平台，做插件

1. 第一阶段：ChatGPT/Claude 的自定义 GPTs 或 Project
2. 第二阶段：浏览器插件，直接读创作者后台数据
3. 第三阶段：API + 客户端

**用户在哪，判镜就在哪。**

## 一个用户的完整旅程

1. 小李发了 15 条视频，2 条破万，不知道为什么
2. 朋友推荐判镜 GPTs，"把链接贴进来"
3. 判镜 40 秒诊断："问题不在选题，在前 10 秒没钩子。高播放 2 条全是冲突性开篇，低播放 13 条全部在做自我介绍"
4. 小李不知道怎么改
5. 判镜给 3 个钩子版本，每个标注情绪机制
6. 小李按建议改，下一条破 5 万
7. 回来做第二轮诊断，此时已有 20 条数据，可以开始做事前预测
8. 小李想转型，"家居内容行吗"
9. 判镜：受众画像不重叠，换赛道等于从零做号
10. 这才是真正有价值的判断
